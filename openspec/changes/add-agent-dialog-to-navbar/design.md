## Context

当前前端主界面使用Element Plus组件库构建，导航栏位于左侧aside区域（Layout.vue）。用户希望在导航栏底部添加一个智能体对话框，用于实现工具调用功能，提升用户交互体验。后端使用Spring Boot + Spring AI框架，支持大模型调用和工具调用。

## Goals / Non-Goals

**Goals:**
- 在导航栏底部添加智能体对话框组件
- 实现消息输入框和发送按钮
- 支持对话框展开/收起切换
- 使用Vue 3 Composition API和Element Plus组件
- 后端实现消息处理和工具调用逻辑
- 集成Spring AI大模型调用能力

**Non-Goals:**
- 对话历史记录展示（不实现）
- 工具调用状态和结果展示（不实现）
- 消息发送和接收逻辑（不实现）
- 自动滚动到最新消息功能（不实现）
- 复杂的工具调用交互（如表单填写）
- 用户登录验证（复用现有逻辑）

## Decisions

### 决策1：组件结构设计
- **选择**: 创建独立的AgentDialog.vue组件，通过插槽或直接引入的方式集成到Layout.vue中
- **理由**: 独立组件便于维护和复用，符合Vue组件化开发最佳实践
- **替代方案**: 直接在Layout.vue中添加对话框代码 → 缺点：代码臃肿，不利于维护

### 决策2：对话框位置
- **选择**: 在导航栏（el-aside）内部底部添加，使用flex布局将对话框固定在底部
- **理由**: 保持导航栏整体布局一致性，用户可以随时访问对话框
- **替代方案**: 在页面其他位置（如右下角）添加浮动对话框 → 缺点：不符合用户"导航栏底部"的需求

### 决策3：消息发送机制
- **选择**: 使用现有request工具类调用后端API
- **理由**: 复用现有技术栈，减少学习成本
- **替代方案**: 使用WebSocket → 缺点：增加复杂度

### 决策4：后端架构设计
- **选择**: 采用Controller → Service → Spring AI的三层架构
- **理由**: 符合项目现有架构风格，便于维护和扩展
- **替代方案**: 直接在Controller中调用大模型 → 缺点：业务逻辑耦合，不利于测试

### 决策5：工具调用实现
- **选择**: 使用Spring AI的Function Calling功能实现工具调用
- **理由**: Spring AI提供了标准化的工具调用接口，与大模型无缝集成
- **替代方案**: 手动实现工具调用逻辑 → 缺点：开发成本高，兼容性差

### 决策6：状态管理
- **选择**: 使用Vue 3 Composition API的ref和reactive管理组件内部状态
- **理由**: 组件状态相对独立，无需引入Vuex或Pinia
- **替代方案**: 使用Pinia全局状态管理 → 缺点：过度设计，增加复杂度

## Risks / Trade-offs

- **风险**: 对话框可能影响导航栏布局 → **缓解**: 使用flex布局确保对话框固定在底部，不影响其他菜单项
- **风险**: 大模型调用超时 → **缓解**: 设置合理的超时时间（180秒），添加错误处理
- **风险**: 工具调用权限问题 → **缓解**: 复用现有Spring Security认证机制
